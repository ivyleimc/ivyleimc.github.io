为什么要写 super(props)

  • super()，执行父类构造函数，初始化 this
  • super(props), 传 props 进去是为了初始化 this.props
1
2

//初始化 this.props, React 内相当于
this.props = props;

不传 props 的话，React 也会在实例化后，初始化 this.props

1
2
3

//React 内相当于
let instance = new myComponent(props);
instance.props = props;

但这样的话，实例化完之前都不能用 this.props 了，所以还是建议传 props

1
2
3
4
5
6

//React 内相当于
constructor(props) {
   super();
   console.log(props); // {name: 'cheng'}
   console.log(this.props); // undefined
}

React 如何区分 Class 和 Function

  • 考虑 Babel 编译后的情况
  • 了解 JavaScript 原型链相关知识
  • class 必须用 new 调用，否则报错
  • 箭头函数不能用 new 调用，否则报错
  • 箭头函数没有 prototype 属性，没有自己的 this
  • 用 new 调用 Function 函数则无法获取原始类型的返回值

React 的做法
在 React.Component 的 prototype 上加 isReactComponent 属性

1
2
3
4
5

//React 内相当于
React.Component.prototype.isReactComponent = {};
// 检查时
class myComponent extends React.Component {...}  
console.log(myComponent.prototype.isReactComponent); // {}

👧 我的思考
一律先直接调用 myComponent()，使用tryCatch包住，报错了就改用 new 调用（现在的 Babel 都考虑到 class 要 new 调用，否侧报错）。
这样做就不优雅，但感觉可行

防止服务器返回 json 文本安全漏洞（浏览器支持 Symbol）

  • 除了做文本转译来防止前端攻击
  • 还在 React 元素上加 $$typeof 属性，防止服务器返回有问题的 json 文本造成的攻击

如果服务器返回这样的文本，会存在问题

let expectedTextButGotJSON = {

为了处理这样的情况，就加了 $$typeof 属性
json 不支持 Symbol 类型，返回的 json 不会含有 Symbol.for(‘react.element’)
React 会检测 element.$$typeof，如果丢失或者无效，会拒绝处理该元素
