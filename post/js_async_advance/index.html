事件循环 1（only 宏任务）

  1. JS 是单线程运行的
  2. 异步要基于回调来实现
  3. event loop 就是异步回调的实现原理
  1. 代码放到 Call Stack 里执行
  2. 有 callback 则放到 Web APIs 等待时机，时机到了放到 callback queue
  3. Call Stack 里的代码执行完了，event loop 启动
  4. 把 callback queue 的 callback 放到 Call Stack 执行
  5. 有 callback 则放到 Web APIs 等待时机，时机到了放到 callback queue
  6. Call Stack 里的代码执行完了，event loop 启动
  7. 如此反复。。。

Promise

Promise 的三个状态

  • pending
  • fulfilled
  • rejected
  • new Promise 里的方法执行了（同步或者异步）resolve()，状态从 pending 变成 fulfilled
  • new Promise 里的方法执行了（同步或者异步）reject()，状态从 pending 变成 rejected
  • new Promise 里的方法报错了（同步），状态也会变成 rejected
  • new Promise 里的方法其他情况, 则一律为 pending

Promise 状态如何变化

  • new Promise 里的方法执行了（同步或者异步）resolve()，状态从 pending 变成 fulfilled
  • new Promise 里的方法执行了（同步或者异步）reject()，状态从 pending 变成 rejected
  • new Promise 里的方法报错了（同步），状态也会变成 rejected
  • new Promise 里的方法其他情况, 则一律为 pending
  • 状态 fulfilled 会触发 then 回调函数
  • 状态 rejected 会触发 catch 回调函数
  • 状态 pending 不会触发 then 和 catch

//执行了 resolve()，状态从 pending 变成 fulfilled
var p1 = new Promise(function(resolve, reject) {
    setTimeout(function() {
        resolve(123);
    });
});
console.log('p1', p1);
setTimeout(function() {console.log('p1 setTimeout', p1);});

//new Promise 里的方法
//执行了 reject()，状态从 pending 变成 rejected
var p2 = new Promise(function(resolve, reject) {
    setTimeout(function() {
        reject(456);
    });
});
console.log('p2', p2);
setTimeout(function() {console.log('p2 setTimeout', p2);});
//
//new Promise 里的方法
//既不执行 resolve() 也不执行 reject()，则状态为 pending
var p3 = new Promise(function(resolve, reject) {
    return 789;
});
console.log('p3', p3);
setTimeout(function() {console.log('p3 setTimeout', p3);});

//状态 fulfilled 会触发 then 回调函数
var p4 = new Promise(function(resolve, reject) {
    setTimeout(function() {
        resolve(123);
    });
}).then(function(con) {
    console.log('p4 then', con);
}).catch(function(err) {
    console.log('p4 catch', err);
});
console.log('p4', p4);
setTimeout(function() {console.log('p4 setTimeout', p4);});
//
//状态 fulfilled 会触发 then 回调函数
//即使 then 回调函数 在 catch 回调函数后面
var p5 = new Promise(function(resolve, reject) {
    setTimeout(function() {
        resolve(123);
    });
}).catch(function(err) {
    console.log('p5 catch', err);
}).then(function(con) {
    console.log('p5 then', con);
});
console.log('p5', p5);
setTimeout(function() {console.log('p5 setTimeout', p5);});
//
//
//状态 rejected 会触发 catch 回调函数
var p6 = new Promise(function(resolve, reject) {
    setTimeout(function() {
        reject(123);
    });
}).then(function(con) {
    console.log('p6 then', con);
}).catch(function(err) {
    console.log('p6 catch', err);
});
console.log('p6', p6);
setTimeout(function() {console.log('p6 setTimeout', p6);});

then/catch 对状态的影响

  • 无论是在 then 还是在 catch 的方法里，报错都会让状态变成 rejected
  • 无论是在 then 还是在 catch 的方法里，只有不报错状态就是 fulfilled

async/await

  • 感觉是 promise 的一颗语法糖
  • 可用同步的语法编写异步的代码

await 后可跟？

  • promise
  • async function
  • 值类型的变量或者字面量

async/await 和 promise 的关系

  • async/await 用同步的语法编写异步的代码
  • 和 promise 不互斥，两者相互想成
  • 执行 async 函数返回的是 promise 对象
  • await 相当于 promise 的 then
  • try…catch 可捕获异常，代替 promise 的 catch
  • 如果 await 后者是个 rejected 状态的 promise 会报错，要用 try…catch 处理

DOM渲染

  • JS 是单线程的
  • DOM 渲染和 JS 是公用一个线程的
  • JS 执行时要留一些时机给 DOM 渲染

事件循环 2（宏任务，DOM渲 ）

  1. 代码放到 Call Stack 里执行
  2. 有 callback 则放到 Web APIs 等待时机，时机到了放到 callback queue
  3. Call Stack 里的代码执行完了
  4. 尝试 DOM 渲染 (不一定每次都会渲染的)
  5. event loop 启动
  6. 把 callback queue 的 callback 放到 Call Stack 执行
  7. 有 callback 则放到 Web APIs 等待时机，时机到了放到 callback queue
  8. Call Stack 里的代码执行完了
  9. 尝试 DOM 渲染 (不一定每次都会渲染的)
  10. event loop 启动
  11. 如此反复。。。

事件循环 3（宏任务，DOM渲染， 微任务）

  • 宏任务：setTimeout, setInterval, Ajax, DOM事件
  • 微任务：Promise, async/await
  • 微任务执行时机比宏任务早

Promise, async/await 是 ES 规范，不是 W3C 规范
所以不会经过 Web APIs，执行时机也就有不同

  • 微任务： DOM 渲染前执行
  • 宏任务： DOM 渲染后执行
  1. 代码放到 Call Stack 里执行
  2. 有 callback 则放到 Web APIs 等待时机，时机到了放到 callback queue
    • 有 promise 则放到微任务队列里
  3. Call Stack 里的代码执行完了
  4. 把微任务里的代码放到 Call Stack 里执行，直到微任务队列里没东西了
  5. 尝试 DOM 渲染 (不一定每次都会渲染的)
  6. event loop 启动
  7. 把 callback queue 的 callback 放到 Call Stack 执行
  8. 有 callback 则放到 Web APIs 等待时机，时机到了放到 callback queue
    • 有 promise 则放到微任务队列里
  9. Call Stack 里的代码执行完了
  10. 把微任务里的代码放到 Call Stack 里执行，直到微任务队列里没东西了
  11. 尝试 DOM 渲染 (不一定每次都会渲染的)
  12. event loop 启动
  13. 如此反复。。。

每次 Call Stack 清空，轮询结束

DOM 事件和 event loop

DOM 事件不是异步，但它和异步一样使用回调，和异步一样基于 event loop
